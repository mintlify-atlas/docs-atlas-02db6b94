## Overview

Log features a powerful full-text search system that allows users to discover public posts across the platform. Built on PostgreSQL's native full-text search capabilities, it provides fast, relevant results ranked by relevance.

## How Search Works

Log's search uses PostgreSQL's `ts_vector` and `ts_query` functions for full-text indexing:

<Steps>
  <Step title="Indexing">
    Post text content is automatically indexed using PostgreSQL's GIN index
  </Step>
  <Step title="Query Processing">
    Search queries are processed and tokenized for matching
  </Step>
  <Step title="Ranking">
    Results are ranked by relevance using `ts_rank` scoring
  </Step>
  <Step title="Filtering">
    Only public posts are included in search results
  </Step>
</Steps>

<Info>
  Search can be enabled or disabled instance-wide by administrators through the `searchStatus` key-value setting.
</Info>

## Database Implementation

### Full-Text Index

The `user_posts` table includes a GIN index for search:

```sql
CREATE INDEX title_search_index ON user_posts 
USING gin(to_tsvector('english', text_data));
```

This index provides:

- **Fast lookups**: Optimized for text search queries
- **Language support**: English language stemming and stop words
- **Automatic updates**: Index stays synchronized with post content

### Search Query

The search endpoint executes this PostgreSQL query:

```sql
SELECT *,
       ts_rank(to_tsvector('english', text_data),
               plainto_tsquery('english', $query)) AS rank
FROM user_posts
WHERE to_tsvector('english', text_data)
      @@ plainto_tsquery('english', $query)
AND status IN ('public')
ORDER BY rank DESC;
```

<Note>
  The `plainto_tsquery` function handles plain text queries, automatically parsing them into proper search terms.
</Note>

```typescript Fetch Search Results
const response = await fetch(
  `/api/data/search?query=${encodeURIComponent("sunset beach")}`
);

console.log(`Found ${data.rows.length} results in ${queryTime}ms`);
```

```typescript React Search Component
import { useState } from "react";

function SearchBar() {
  const [query, setQuery] = useState("");
  const [results, setResults] = useState([]);

  const handleSearch = async () => {
    const response = await fetch(
      `/api/data/search?query=${encodeURIComponent(query)}`
    );
    const { data } = await response.json();
    setResults(data.rows);
  };

  return (
    <div>
      <input 
        value={query} 
        onChange={(e) => setQuery(e.target.value)}
        onKeyPress={(e) => e.key === "Enter" && handleSearch()}
      />
      <button onClick={handleSearch}>Search</button>
    </div>
  );
}
```
</CodeGroup>

## Search Features

### Relevance Ranking

Results are ordered by relevance score:

- **Higher scores** = better matches
- **Factors considered**:
  - Term frequency (how often search terms appear)
  - Document length normalization
  - Term position and proximity

<Card title="Ranking Example">
  A post with "sunset" mentioned 5 times will rank higher than one with a single mention, all else being equal.
</Card>

### Query Processing

The search engine handles:

- **Stemming**: "running" matches "run", "runs", "runner"
- **Stop words**: Common words like "the", "a", "is" are ignored
- **Case insensitive**: "SUNSET" and "sunset" produce identical results
- **Phrase queries**: Search for exact phrases using quotes (future enhancement)

### Performance Metrics

Every search response includes timing data:

```typescript
{
  "queryTime": 15.3  // Time in milliseconds
}
```

This helps monitor search performance and identify optimization opportunities.

## Instance Configuration

### Enabling/Disabling Search

Administrators can control search availability:

```typescript
// Check search status from kv_data table
SELECT value FROM kv_data WHERE key = 'searchStatus';

// Returns: true (enabled) or false (disabled)
```

When disabled, the API returns:

```json
{
  "success": false,
  "msg": "Search functionality is currently disabled.",
  "data": [],
  "disabled": true
}
```

<Info>
  Disabling search may be useful during maintenance, for privacy-focused instances, or to reduce server load.
</Info>

## Technical Details

### PostgreSQL Configuration

For optimal search performance:

```sql
-- Ensure proper text search configuration
SHOW default_text_search_config;
-- Should return: pg_catalog.english

-- Increase shared_buffers for better index performance
-- In postgresql.conf:
shared_buffers = 256MB  -- Adjust based on available RAM
```

### Index Statistics

```sql
-- Check index usage
SELECT 
  schemaname,
  tablename,
  indexname,
  idx_scan,
  idx_tup_read,
  idx_tup_fetch
FROM pg_stat_user_indexes
WHERE indexname = 'title_search_index';
```

<Tip>
  If `idx_scan` is low, the search feature may not be heavily used. If it's high but queries are slow, consider database tuning.
</Tip>
